Getting Started

Quick Start — Test a Conversion in 60 Seconds

The fastest way to see the converter in action. No setup required — just a browser.

🌐

Live App: ca-pdftohtml-frontend.blackplant-e84b1473.eastus.azurecontainerapps.io

Upload & Convert (Browser)

  1. Open the frontend app in your browser
  2. Drag & drop a PDF, DOCX, or PPTX file onto the upload zone (or click to browse)
  3. You'll be redirected to the Dashboard automatically
  4. Watch the status change: pending → processing → completed
  5. Click Preview to see the accessible HTML inline, or Download to save it

Upload & Convert (CLI / curl)

Quick Conversion Test 
# Set the API base URL
$ API="https://ca-pdftohtml-api.blackplant-e84b1473.eastus.azurecontainerapps.io"

# Step 1: Get a SAS upload token
$ curl -s -X POST "$API/api/upload/sas-token" \
  -H "Content-Type: application/json" \
  -d '{"filename":"test.pdf","content_type":"application/pdf","size_bytes":12345}'

# Step 2: Upload your file to the returned upload_url
$ curl -X PUT "<upload_url_from_step_1>" \
  -H "x-ms-blob-type: BlockBlob" \
  --data-binary @your-file.pdf

# Step 3: Poll status until "completed"
$ curl -s "$API/api/documents/status"

# Step 4: Download the converted HTML
$ curl -s "$API/api/documents/<document_id>/download"

Section 01

System Architecture

The NCDIT WCAG Document Converter is a containerized document processing pipeline deployed on Azure Container Apps. Files uploaded through the Next.js frontend are stored in Azure Blob Storage. Event Grid routes blob-created events to a Storage Queue, which a KEDA-scaled worker container consumes to extract content, apply WCAG compliance rules, and generate accessible HTML output.

Section 02

Deployment Guide

Prerequisites

  • Azure subscription with Contributor access
  • Azure CLI installed and authenticated (az login)
  • Docker (for local builds) or Azure Container Registry access
  • Resource group rg-pdf-to-html with Container Apps environment configured

Architecture

Three Container Apps share a single environment. The API and Worker use the same Docker image (different entrypoints). The Frontend has its own Next.js image.

Build & Deploy (CI/CD)

Push to main triggers .github/workflows/deploy-aca.yml. It builds images via ACR Tasks and deploys all three apps automatically.

Manual Deployment 
# Build backend image via ACR
$ az acr build --registry crpdftohtml \
  --image pdf-to-html-api:latest \
  --file Dockerfile.backend .

# Deploy API container
$ az containerapp update --name ca-pdftohtml-api \
  --resource-group rg-pdf-to-html \
  --image crpdftohtml.azurecr.io/pdf-to-html-api:latest

# Deploy Worker container (same image, different command)
$ az containerapp update --name ca-pdftohtml-worker \
  --resource-group rg-pdf-to-html \
  --image crpdftohtml.azurecr.io/pdf-to-html-api:latest

# Deactivate old revisions
$ az containerapp revision list --name ca-pdftohtml-api \
  --resource-group rg-pdf-to-html -o table

Frontend Deployment

Frontend Build & Deploy 
# Get the API FQDN (needed at build time!)
$ API_FQDN=$(az containerapp show --name ca-pdftohtml-api \
  --resource-group rg-pdf-to-html --query properties.configuration.ingress.fqdn -o tsv)

# Build frontend with BACKEND_URL baked in
$ az acr build --registry crpdftohtml \
  --image pdf-to-html-frontend:latest \
  --build-arg BACKEND_URL=https://$API_FQDN \
  --file frontend/Dockerfile ./frontend

# Deploy frontend container
$ az containerapp update --name ca-pdftohtml-frontend \
  --resource-group rg-pdf-to-html \
  --image crpdftohtml.azurecr.io/pdf-to-html-frontend:latest
⚠️

Critical: The BACKEND_URL is baked into the frontend at build time (Next.js standalone mode). Runtime env vars have NO effect. Always pass --build-arg BACKEND_URL=https://<api-fqdn> during image build.

✅

Verify Deployment: Hit https://ca-pdftohtml-api.../health for the API, check the frontend loads, and upload a test PDF to confirm the worker is processing.

Section 03

Configuration Reference

Environment variables are set on each Container App via az containerapp update --set-env-vars or in the Bicep templates under infra/modules/container-apps.bicep.

Variable Required Default Description
AZURE_STORAGE_CONNECTION_STRING ✅ UseDevelopmentStorage=true Azure Blob/Queue Storage connection string
DOCUMENT_INTELLIGENCE_ENDPOINT ✅ — Azure Document Intelligence (OCR) endpoint URL
DOCUMENT_INTELLIGENCE_KEY ✅ — API key for Document Intelligence
INPUT_CONTAINER ❌ files Blob container for uploaded documents
OUTPUT_CONTAINER ❌ converted Blob container for conversion results
QUEUE_NAME ❌ conversion-jobs Azure Storage Queue for conversion tasks
PORT ❌ 8000 FastAPI server listen port
LOG_LEVEL ❌ INFO Logging level (DEBUG, INFO, WARNING, ERROR)
ALLOWED_ORIGINS ❌ (frontend URL) CORS allowed origins (comma-separated)
⚠️

Secrets Management: Never commit DOCUMENT_INTELLIGENCE_KEY or connection strings to source control. Set secrets via az containerapp update --set-env-vars "KEY=secretref:key-name" or Azure Key Vault references.

Section 04

API Endpoints

The FastAPI backend (app/main.py) exposes these endpoints. No auth keys required — access is controlled by Container Apps ingress configuration.

Method Path Description
GET /health Liveness probe — storage & queue connectivity check
GET /ready Readiness probe — returns 503 if storage unreachable
POST /api/upload/sas-token Generate SAS token for direct browser-to-blob upload
GET /api/documents/status Get status of all documents (or single with ?document_id=)
GET /api/documents/{id}/download Get SAS-signed download URLs for converted HTML + assets
DELETE /api/documents/{id} Delete a single document (input + output blobs)
DELETE /api/documents Bulk delete all documents

Processing Pipeline

Conversion is asynchronous — the worker (app/worker.py) polls the conversion-jobs Storage Queue every 2 seconds. Event Grid fires on BlobCreated events in files/ container (filtered by contentLength > 0) and delivers to the queue.

Section 05

Operations & Monitoring

Health Checks

  • API liveness: GET /health — checks storage & queue connectivity
  • API readiness: GET /ready — returns 503 if storage unreachable
  • Worker: Logs every 2s poll cycle — check container logs for heartbeat
  • Frontend: Hit the frontend URL — should return 200

Key Metrics to Monitor

  • Container restart count (API, Worker, Frontend)
  • Queue depth — messages in conversion-jobs
  • Conversion duration — logged as processing_time_ms in blob metadata
  • OCR API call count and latency
  • Error rate — documents with status=failed
  • Pages flagged for review (has_review_flags=true)

Log Access

Container Logs 
# Stream API logs
$ az containerapp logs show --name ca-pdftohtml-api \
  --resource-group rg-pdf-to-html --follow

# Stream Worker logs (where conversion happens)
$ az containerapp logs show --name ca-pdftohtml-worker \
  --resource-group rg-pdf-to-html --follow

# Get system-level logs for a container
$ az containerapp logs show --name ca-pdftohtml-worker \
  --resource-group rg-pdf-to-html --type system

# Check queue depth (pending conversions)
$ az storage message peek --queue-name conversion-jobs \
  --account-name stpdftohtml331ef3 --num-messages 32
ℹ️

Tip: Azure SDK logging is suppressed in the worker (logging.WARNING level). App-level messages at INFO and above will appear. Set LOG_LEVEL=DEBUG for verbose output during troubleshooting.

Section 06

Troubleshooting

Common issues and their resolutions. Always check worker container logs first.

Symptom Likely Cause Resolution
All conversions fail immediately Code error in worker (e.g. NameError) Check worker logs: az containerapp logs show -n ca-pdftohtml-worker
Documents stuck at "pending" Event Grid subscription misconfigured or queue empty Check Event Grid subscription exists, verify conversion-jobs queue has messages
OCR returns empty / pages flagged for review DOCUMENT_INTELLIGENCE_KEY expired (401) Regenerate key and update worker env var
Preview returns 404 Output path mismatch — blob metadata missing output_path Check blob metadata in storage explorer; re-upload to trigger re-conversion
Download "failed to fetch" in browser Blob storage CORS missing GET method Update CORS: az storage cors add ... --methods GET PUT OPTIONS HEAD
"Unable to reach the server" on frontend BACKEND_URL not set at build time Rebuild frontend image with --build-arg BACKEND_URL=https://<api-fqdn>
Upload fails / SAS token error Storage CORS or SAS token expired Check storage CORS allows PUT from frontend origin; verify SAS generation
"Password-protected" error Encrypted document uploaded User must remove password before uploading
Worker container keeps restarting Connection string wrong or missing Verify AZURE_STORAGE_CONNECTION_STRING on worker container

Section 07

Incident Response

Severity Levels

P1 — Critical (Service Down)

  1. Check container status: az containerapp show --name ca-pdftohtml-api -g rg-pdf-to-html --query properties.runningStatus
  2. Check container logs for crash loops or OOM kills
  3. Rollback to previous revision: az containerapp revision activate --revision <prev-revision>
  4. If storage issue, check connection string and storage account health
🔴

P1 Alert: Notify on-call engineer immediately. Document all troubleshooting steps in incident ticket. Target resolution time: 1 hour.

P2 — High (Degraded Performance)

  1. Check worker log for processing delays or OCR timeouts
  2. Check queue depth — if messages are piling up, worker may be stuck
  3. Restart worker: az containerapp revision restart --name ca-pdftohtml-worker -g rg-pdf-to-html --revision <revision>
  4. Scale up if needed: update Container Apps min/max replicas in Bicep

P3 — Medium (Individual Failures)

  1. Check document-specific error in blob metadata (error_message field)
  2. Use the Retry button in the dashboard to re-upload and re-trigger conversion
  3. Check if document is corrupted, password-protected, or unsupported format

Section 08

Security & Compliance

Compliance Overview

  • WCAG 2.1 AA Compliance: All HTML output validated against 7 server-side checks
  • File Validation: Only PDF, DOCX, PPTX accepted (max 100MB)
  • Encrypted Documents: Automatically rejected with user-friendly error
  • SAS Tokens: Short-lived (1 hour), scoped to specific blob
  • No PII in Logs: Document content never logged; Azure SDK logging suppressed
  • Container Security: Non-root user (appuser:1001) in all containers
  • Access Control: Container Apps ingress controls, no function keys
  • Data Residency: All processing in Azure East US

Key Rotation Procedure

Key Rotation 
# Rotate Document Intelligence keys
$ az cognitiveservices account keys regenerate \
  --name <doc-intelligence-name> \
  --resource-group rg-pdf-to-html --key-name key1

# Update worker container env var
$ az containerapp update --name ca-pdftohtml-worker \
  --resource-group rg-pdf-to-html \
  --set-env-vars "DOCUMENT_INTELLIGENCE_KEY=<new-key>"
⚠️

Security Reminder: Rotate all keys quarterly. Use Azure Key Vault for production secrets. Enable soft delete on blob storage (14-day retention) to recover accidentally deleted files.
